# 知识树生成与可视化工具：设计文档

## Opening：背景介绍

对于学生、研究人员或任何需要快速入门一个新知识领域的人来说，整理学习路线和分清概念之间的层级关系往往是一项非常耗时的工作。在之前，我们可能会通过搜索引擎、百科和学术论文来拼凑出一张“认知地图”，但这个过程往往是分散并缺乏结构的。

本项目旨在提供一个轻量级的本地工具。用户只需输入一个核心关键词（例如：“强化学习”、“地中海饮食”、“唐朝职官制度”），系统便能借助大语言模型的理解能力，自动生成一棵结构化的“知识树”。这棵树不仅包含核心定义，还会将关键的子领域、重要概念以及相关的经典学术论文按照一定层级联系起来。为了降低使用门槛，我们把项目打包成了单文件可执行程序。

## Challenge：所要解决的问题

在构建该工具时，我们遇到了几个极具挑战性的问题：

1.  **生成内容的结构化与不确定性**：大语言模型虽然知识渊博，但它的输出是自然语言流，而非程序可直接渲染的树状数据结构。如何确保模型返回的内容是前端画布，是能准确解析的、格式严格的 JSON 树？如果模型输出格式错误或网络超时，工具应该如何产生输出以保证用户体验不降级？

2.  **本地数据的可靠性与可扩展性**：用户保存的知识树需要持久化。如果使用复杂的数据库（如 SQLite），虽然功能强大，但会显著增加打包体积和部署复杂度，且不便于用户手动查看或备份数据。要如何在“功能完备”与“简单透明”之间做取舍？

3.  **前端渲染的性能与交互流畅度**：知识树可能包含数十甚至上百个节点。在网页上绘制这种可缩放、可拖拽的关系图，既要保证动画流畅，又要支持点击展开、查看论文详情等交互操作。该选择什么图形渲染工具才能在不引入大型框架的前提下保持代码简洁？

4.  **密钥安全与分发便利性的矛盾**：工具需要调用 DeepSeek API，这涉及 API Key 的配置。如果将密钥硬编码打包进 exe 文件，不仅极不安全，还会导致密钥泄漏后全员不可用。如何在让普通用户双击 exe 就能运行的同时，又能安全地，简便的引导他们配置自己的密钥？

5.  **论文来源是否可靠与版权问题**：工具需要调用大语言模型进行论文的筛选与输出，但这个过程往往会出现“AI幻觉”，生成的论文链接可能是不可靠甚至不存在的。于是我们采用arxiv，openalex和semantic——scholar三个权威的论文平台的合法api接口与大语言模型的筛选功能进行对接，使得关联论文的来源是权威可靠的，不存在非法爬虫问题。

## Action：执行的实验/设计/取舍

针对上述挑战，我们在设计阶段采取了以下关键决策与架构权衡：

### 1. 后端容错与“回退树”机制
为了应对 LLM 输出格式不稳定的问题，我们设计了**双重保障策略**。
- **严格提示词约束**：在发送给 DeepSeek 的 Prompt 中，我们明确要求仅返回 JSON 格式，并提供了详细的 Schema 示例。
- **回退树（Fallback Tree）设计**：考虑到网络波动或 API 余额不足的情况，我们在后端内置了一棵硬编码的“本地示例树”。一旦解析 LLM 响应失败或网络请求异常，系统会**自动降级**返回这棵本地树。**取舍理由**：与其向用户展示一个冰冷的“500 Internal Server Error”，不如展示一棵虽然与搜索词无关、但结构完整的示例树。这能让用户第一时间理解工具的预览效果形态。

### 2. 以文件系统代替数据库的持久化设计
我们没有引入任何 SQL 或 ORM 库，而是选择使用单一的 `storage.json` 文件作为数据层（通过 `restore.py` 读写）。
- **设计思路**：知识树本身就是树形结构，JSON 是其最自然的表达方式。用户可以通过记事本直接打开该文件进行批量修改或备份。
- **取舍理由**：放弃数据库的复杂查询能力（如多条件关联检索），换取**零配置、高透明度、极简打包体积**。对于个人知识库工具而言，几十棵树的列表检索完全可以通过 Python 内存遍历高效完成，无需数据库索引。

### 3. 前端可视化方案的演进：从 ECharts 到原生 Canvas
在`script.js` 中留有 ECharts 的语句，但随着开发流程的深入，目前主逻辑已切换至**内联 JavaScript 配合原生 Canvas**。
- **设计演变与反思**：最初尝试使用 ECharts Tree 图表是为了快速出图。但很快发现，ECharts 的树图布局算法在节点数量多且需要频繁交互（如点击展开子节点、拖拽画布）时，其默认的事件监听机制会与我们的定制需求产生冲突，导致代码变得臃肿且非常难以调试。
- **最终方案（Action）**：于是，我们回归到原生 Canvas 自主绘制。我们实现了简单的**力导向布局算法变体**来计算节点坐标。虽然代码量比引入图表库稍多，但它带来了完全的渲染控制权。现在，节点的蓝色“+”按钮、点击事件、平滑缩放完全由我们自己的轻量逻辑控制，性能损耗极低，且不再依赖庞大的第三方库网络加载。

### 4. 打包策略与配置分离
为了解决分发便利性与安全性的矛盾，我们在 `PyInstaller` 打包配置（`AI_Knowledge_Tree.spec`）中明确将 `.env` 文件排除在打包内容之外。
- **设计思路**：程序启动时，`app.py` 会优先读取同级目录下的 `.env` 文件。如果文件不存在或 Key 为空，程序界面不会崩溃，而是通过前端弹窗提醒用户：“未检测到 API Key，请复制 .env.example 为 .env 并填入你的密钥。”
- **安全性收益**：生成的 `exe` 文件不包含任何敏感凭证，可以放心分享给同事或上传至网盘。用户只需维护自己的环境变量文件。

## Resolution：结果

经过上述设计迭代，最终交付的是一款**单文件、开箱即用、数据自主可控**的知识管理轻应用。

1.  **用户体验闭环**：
    -   **启动**：双击运行，看到简洁的画布。
    -   **生成**：输入关键词，无论联网与否都能获得一棵结构完整的“知识骨架”（联网则获取 AI 定制树，断网则获取本地示例树）。
    -   **自主修整**：用户可以通过点击节点旁的“+”号，利用 AI 对局部知识点进行**深度扩展**；若 AI 给出的论文链接缺失，`paper_enrich.py` 会在后台尝试补全元数据。
    -   **持久化**：所有修改一键保存至肉眼可读的 `storage.json`。

2.  **开发与维护友好**：代码结构清晰分离。`backend/` 专注数据逻辑与 AI 通信，`frontend/` 专注交互绘制。得益于 API 文档（`DEVELOPMENT.md`）的完整性，未来的维护者可以快速理解接口定义。

3.  **健壮性提升**：通过引入“回退树”和“配置外部化”，系统在最恶劣的环境下（无网、无密钥）依然能够作为一款离线的思维导图编辑器正常工作，避免了工具因外部依赖失效而完全瘫痪的窘境。

本项目证明了，在不增加用户认知负担的前提下，结合大模型 API 的轻量级本地桌面软件依然能提供强大且安全的知识辅助体验。
